## Summary of Differences

### 1. Frontmatter — Description Field

**Original:** A single sentence: `"Cross-document consistency reviewer. Reads specs, designs, and DIPs cold — without conversation context — to catch contradictions, mismatches, and stale references that authors miss. Use after writing or revising any planning artifact."`

**Generated:** A multi-paragraph block with three worked examples showing when a calling agent should invoke this agent and what to say. The description field in agent frontmatter is a routing hint — the platform uses it to decide when to surface this agent. The original is a concise dispatch signal. The generated version embeds 15+ lines of scenario scripting inside what is functionally a one-line field.

**Assessment:** Degrades. The description field is consumed by the routing layer, not by the agent itself. Stuffing worked examples into it bloats the field without adding anything the agent can act on. The examples also script the *calling agent's* response language ("I've updated the design document. Now let me use the fresh-eyes-reviewer agent..."), which is outside this agent's scope entirely. This content belongs in the calling agent's instructions or a tool-use guide, not here.

### 2. Frontmatter — Tools Field

**Original:** `tools: Read, Grep, Glob, Write`

**Generated:** Field is absent.

**Assessment:** Degrades. The tools field constrains which tools the agent is permitted to use. Without it, the agent inherits whatever the platform default is. For a cold-read reviewer, the constraint is intentional: it should only read and write, not execute code, fetch URLs, or call APIs. Removing the constraint makes the agent's blast radius unpredictable.

### 3. Frontmatter — Model Field

**Original:** `model: inherit`

**Generated:** `model: sonnet`

**Assessment:** Neutral on correctness, but a meaningful semantic change. `inherit` means the agent uses whatever model the parent conversation is using. `sonnet` hardcodes a specific model tier. The original design intent was presumably to avoid pinning the agent to a model that may be downgraded or renamed. If the project moves to a newer model, the generated version silently stays on Sonnet while the original would follow. Neither is wrong, but the original's choice is deliberate.

### 8. Reasoning Approach vs. Procedure

**Original:** `## Reasoning Approach` — five numbered steps, then two bullet lists (what to look for, what to ignore).

**Assessment:** Improves on specificity. The generated version's cross-reference checklist is more actionable:

## Recommendations

1. **Restore the `tools` field.** The original's explicit tool constraint is intentional and should be preserved. A cold-read reviewer should not have access to web fetch, code execution, or API tools.

2. **Revert the `description` field to a concise routing hint.** The worked examples in the generated description belong in a separate calling-agent guide, not in the frontmatter of this agent. If the goal is to help the main assistant know when to invoke this agent, that guidance belongs in a skills/commands file, or in the calling agent's system prompt.

3. **Remove the injected memory system documentation from the stored file.** Everything from `# Persistent Agent Memory` onward is platform boilerplate that will be injected at runtime. Keeping it in the stored file creates duplication and includes a hardcoded path that breaks portability.

4. **Consider keeping `model: inherit`** unless there is a specific reason to pin Sonnet. Portability and forward-compatibility favor `inherit`.

5. **Carry forward the improvements** from the generated version — specifically: Project Context section, the more granular cross-reference checklist, the embedded self-evaluation in the output template, the filename convention, and the Critical Rules section. These are genuine improvements.
